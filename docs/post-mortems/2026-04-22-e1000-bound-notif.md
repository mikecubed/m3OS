# Post-mortem: e1000 bound-notification deadlock (SSH-over-e1000 hang)

**Incident:** SSH to the guest over `--device e1000` completed the TCP handshake and
then hung indefinitely — sshd never received the client's SSH version string and never
sent the server banner.
**Status:** Resolved 2026-04-22.
**Severity:** High — blocked the primary sustained-networking acceptance test for the
ring-3 e1000 driver; any interactive SSH session over e1000 was unresponsive.
**Owners:** Kernel IPC (bound-notification primitive), ring-3 e1000 driver.
**Fix commit:** Phase 55c Track F (e1000 `run_io_loop` collapse) + Track B (`sys_notif_bind`
+ `ipc_recv_msg` wake-kind extension).
**Adjacent closures:** R1 (userspace EAGAIN surface for driver restart — Track G/H) and
R2 (IOMMU BAR identity mapping under `--iommu` — Track C/D) closed in the same phase.

## Summary

The ring-3 e1000 driver's main loop in `userspace/drivers/e1000/src/io.rs::run_io_loop`
alternately blocked on two distinct kernel primitives — `IrqNotification::wait()` (on a
`Notification`) and `NetServer::handle_next()` (on an `Endpoint`) — with no mechanism to
service the other source while parked on either. The two wake sources were serialized,
not multiplexed, so an IRQ arriving while the driver was parked on the endpoint silently
piled up as a pending notification bit. The endpoint recv never returned; the RX
descriptor ring was never drained; the kernel TCP stack never saw the incoming frame.

The fix adds a bound-notification primitive following the seL4 model: one `Notification`
can be bound to a TCB, and `ipc_recv_msg(ep_cap)` checks the bound notification's pending
bits atomically before parking on the endpoint. Either wake source — a peer IPC send or a
notification signal from the IRQ handler — unblocks the recv with a typed `WakeKind` tag.
The driver's loop collapses to a single `recv()` per iteration; the two-stage
`irq.wait()` / `handle_next()` alternation is eliminated.

## Root Cause

**The ring-3 driver main loop alternately blocked on `IrqNotification::wait` and
`Endpoint::recv`; the two wake sources were serialized, not multiplexed.** Whichever
source the driver was currently parked on could not be woken by the other, so an IRQ
signaling a new RX frame was silently dropped while the driver waited for a TX request
on the endpoint.

Specific sequence:

1. TCP handshake completes. Client's SYN IRQs the NIC; driver wakes from `irq.wait()`,
   drains the RX frame, falls through to `handle_next()`. Kernel TCP posts SYN-ACK;
   `handle_next()` returns; ACK frame goes out. Same for the client's ACK.
2. Driver re-enters `handle_next()` — blocks inside `ipc_recv_msg` waiting for the next
   TX request from the kernel net stack.
3. Client sends its SSH version string. The e1000 NIC receives the frame, raises IRQ 11.
   The kernel sets the RX bit on the driver's notification object.
4. **The driver is not in `irq.wait()`** — it is parked inside `ipc_recv_msg`. The
   notification bit remains set; the NIC's RX descriptor stays undrained.
5. The kernel TCP stack never sees the incoming segment. `sshd` never reads it. No TX
   request is ever queued to the endpoint. `handle_next()` blocks forever. **Deadlock.**

VirtIO-net did not exhibit this because it is still an in-kernel driver — no main-loop
serialization problem. NVMe did not exhibit this because block devices are purely
request-response: the driver polls for completion inside the request handler, so
`handle_next()` alone suffices.

## Why Phase 55b Did Not Catch It

Phase 55b's acceptance test for the e1000 driver is `device-smoke --device e1000`. That
test exercises a single one-shot ICMP echo request and reply — one RX frame, one TX
frame, then done. The smoke test completes in the TCP handshake window, before the
post-handshake quiescent state where the driver parks on the endpoint. The `device-smoke`
path never reaches the three-way-after-handshake sequence that triggers the deadlock.

**SSH is the first sustained bidirectional workload.** SSH involves a version exchange
(two frames in each direction) plus key-exchange (many frames). The first incoming frame
after the driver has committed to the endpoint's recv is guaranteed to exercise the
deadlock. Phase 55b's one-shot smoke never created that condition.

## Fix Applied

**Bound notifications per the seL4 model** — Phase 55c Tracks A, B, E, and F:

- **Track A** (`kernel-core/src/ipc/bound_notif.rs`): pure-logic state model for
  binding, unbinding, signal, and recv. Property-tested for arbitrary bind/signal/send
  interleavings.
- **Track B** (`kernel/src/ipc/notification.rs`, `kernel/src/ipc/endpoint.rs`,
  `kernel/src/ipc/mod.rs`): `BOUND_TCB[MAX_NOTIFS]` array (lock-free, ISR-safe),
  `sys_notif_bind` syscall, `ipc_recv_msg` extended with a `WakeKind` return channel.
- **Track E** (`userspace/lib/driver_runtime/src/ipc/mod.rs`): `RecvResult` enum
  (`Message(RecvFrame)` | `Notification { bits: u64 }`); `IrqNotification::bind_to_endpoint`
  helper.
- **Track F** (`userspace/drivers/e1000/src/io.rs`): `subscribe_and_bind` issues
  `sys_notif_bind(irq_notif_cap, endpoint_cap)` during init; `run_io_loop` collapses to
  a single `endpoint.recv()` per iteration that returns `RecvResult`, eliminating the
  standalone `irq.wait()` path entirely.

## Confirmation

SSH banner arrives within 5 s after `cargo xtask run --device e1000`. Verified by
`scripts/ssh_e1000_banner_check.sh` (introduced in Track I). The script connects to
`root@localhost -p2222`, waits for the server version string, and asserts the banner
arrives before a 5-second timeout.

## Adjacent Closures

Two residuals from `docs/appendix/phase-55b-residuals.md` were closed in the same phase:

- **R1 — Userspace EAGAIN visibility (Track G/H):** `RemoteNic::send_frame` in
  `kernel/src/net/remote.rs` already returned `NetDriverError::DriverRestarting` to
  kernel-side callers, but the userspace send path never routed through it. Track G
  wired the `sys_net_send` (or `sys_sendto` extension) through `RemoteNic` and mapped
  `DriverRestarting` to `-EAGAIN` via the existing `net_error_to_neg_errno`. Track H
  extended `userspace/e1000-crash-smoke` to assert the EAGAIN is observable and removed
  the `#[ignore]` from `qemu_e1000_kill_mid_send_returns_driver_restarting_then_icmp_echo_succeeds`.

- **R2 — IOMMU BAR identity mapping (Track C/D):** `cargo xtask device-smoke --device
  {nvme,e1000} --iommu` timed out at the driver bring-up step because the IOMMU domain
  had no identity-mapped entry for the device's MMIO BARs. Track C added
  `kernel-core/src/iommu/bar_coverage.rs` (a pure-logic invariant and host tests). Track D
  extended the VT-d and AMD-Vi per-device domain setup to insert identity-mapped 4 KiB
  pages covering each BAR at `sys_device_claim` time. Device-smoke under `--iommu` now
  passes end-to-end.
