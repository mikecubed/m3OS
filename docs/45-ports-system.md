# Ports and Package System

**Aligned Roadmap Phase:** Phase 45
**Status:** Complete
**Source Ref:** phase-45

## Overview

Phase 45 adds a BSD-style ports system for building and installing third-party
software from source inside m3OS. Each port is a directory containing a Portfile
(metadata), a Makefile (build rules), bundled source code, and optional patches.
The `port` shell command automates the full lifecycle: dependency resolution,
source extraction, compilation with TCC, installation to `/usr/local/`, and
package tracking with file manifests for clean removal.

## What This Doc Covers

- The Portfile metadata format and Makefile target contract
- How the `port` shell command drives the build lifecycle
- Dependency resolution and package tracking via flat-file databases
- How xtask bundles the ports tree and source into the disk image
- How to create a new port

## Core Implementation

### Portfile Format

Each port's `Portfile` uses shell variable syntax so the `port` command can
source it directly with `. /usr/ports/lang/lua/Portfile`:

```sh
NAME=lua
VERSION=5.4.7
DESCRIPTION="Lightweight scripting language"
CATEGORY=lang
DEPS=
MAINTAINER=m3os
```

Required fields: `NAME`, `VERSION`, `DESCRIPTION`, `CATEGORY`, `DEPS` (space-
separated dependency names, or empty). Optional fields: `URL`, `SHA256`,
`MAINTAINER`.

### Makefile Target Contract

Every port implements five standard make targets:

| Target | Behavior |
|---|---|
| `fetch` | Copy bundled source from `src/` to `work/` |
| `patch` | Apply patches from `patches/*.patch` via `patch -p1` |
| `build` | Compile with TCC (e.g., `tcc -o lua *.c`) |
| `install` | Copy outputs to `$(PREFIX)` (default `/usr/local`) |
| `clean` | Remove the `work/` build directory |

All Makefiles use `PORTDIR`, `PREFIX`, and `WORKDIR` variables for
path-independent operation. The `PREFIX` variable defaults to `/usr/local` and
can be overridden at install time.

### The `port` Command (`/usr/bin/port`)

A shell script installed at `/usr/bin/port` with five subcommands:

**`port list`** â€” Scans `/usr/ports/` with `find` for all Portfiles, sources
each one, and prints a tabular list with name, version, description, and
installed status.

**`port info <name>`** â€” Locates a port by scanning all categories, sources its
Portfile, and prints all metadata fields plus install status.

**`port install <name>`** â€” The core subcommand. Workflow:
1. Check if already installed (skip if so).
2. Locate the port directory by name via `find_port`.
3. Source the Portfile and resolve dependencies (`resolve_deps`).
4. Snapshot files under `$PREFIX` before install.
5. Run `make fetch`, `make patch`, `make build`, `make install` in sequence.
6. Snapshot files after install and diff to generate the manifest.
7. Record the install in `/var/db/ports/installed`.

Any failure at a make target aborts the install with a clear error message.

**`port remove <name>`** â€” Reads the file manifest at
`/var/db/ports/<name>.manifest`, deletes each listed file, removes the manifest,
and updates the installed database. Warns if other installed ports depend on the
port being removed.

**`port clean [name]`** â€” Runs `make clean` for a specific port or all ports,
removing `work/` build directories.

### Dependency Resolution

The `resolve_deps` function reads the `DEPS` field from a port's Portfile.
For each listed dependency that is not yet installed (checked via `is_installed`
which greps `/var/db/ports/installed`), it recursively calls `cmd_install` to
build and install the dependency first. Already-installed dependencies are
skipped.

### Package Tracking

Two flat-file databases at `/var/db/ports/`:

- **`installed`** â€” One line per installed port: `name version date`. Written by
  `track_install` after each successful install.
- **`<name>.manifest`** â€” One file path per line, listing every file installed by
  the port. Generated by `generate_manifest`, which compares `find $PREFIX -type f`
  before and after `make install`. Used by `port remove` to delete exactly the
  right files.

### xtask Integration

Three functions in `xtask/src/main.rs` handle ports:

**`fetch_lua_source` / `fetch_zlib_source`** â€” Download Lua 5.4.7 and zlib 1.3.1
source tarballs at build time via `curl`, extract them to
`target/ports-src/<category>/<name>/src/`, and cache the result. Other ports
(bc, sbase, mandoc, minizip) ship self-contained source in-repo.

**`populate_ports_tree`** â€” Called from `create_data_disk()` after
`populate_demo_project()`. Mirrors the host `ports/` directory into `/usr/ports/`
on the ext2 partition via `debugfs`, overlaying downloaded source from
`target/ports-src/`. Also:
- Installs `port.sh` as `/usr/bin/port` with execute permissions.
- Creates `/usr/local/{bin,lib,include}` for port installs.
- Creates `/var/db/ports/` for the package database.

### The Ports Collection

| Port | Category | Type | Dependencies |
|---|---|---|---|
| lua | lang | Scripting language | none |
| zlib | lib | Compression library | none |
| bc | math | Calculator | none |
| sbase | core | 15 Unix utilities | none |
| mandoc | doc | Man page formatter | none |
| minizip | util | Compression test tool | zlib |

Lua and zlib are real third-party code fetched at build time. bc, sbase,
mandoc, and minizip are self-contained C implementations bundled in the
repository under `ports/<category>/<name>/src/`.

## How This Phase Differs From Later Work

- This phase provides source-based package management only. Binary package
  distribution (precompiled packages) is deferred.
- All port sources are bundled in the disk image. Network fetching is deferred
  until an HTTP client is ported (Phase 45 stretch goal or later).
- Dependency resolution is simple (linear scan, no version constraints). Version
  conflict resolution and upgrade handling are deferred.
- No package signing or integrity verification.
